이번 포스팅은 `DaC` 관련 개인적인 경험을 스크랩 해 보았다.

먼저, [Write the Docs](https://www.writethedocs.org/guide/docs-as-code)에서는 `DaC`를 이용하면 다음과 같은 장점이 있다고 설명한다.
- Writers integrate better with development teams.
- Developers will often write a first draft of documentation.
- You can block merging of new features if they don’t include documentation, which incentivizes developers to write about features while they are fresh.

내가 DaC를 가장 많이 사용하는 때라면 아마 프로젝트에서 *README*나 *Wiki*를 작성할 때, 간단한 Syntax 사용을 위한 문서로 Markdown을 적용할 때. 혹은 설계문서나 기타 개발 도큐먼트를 작성할 때인 것 같다.

나는 간단하게 사용하고 싶은 경우에는 [MkDocs](https://www.mkdocs.org)나 [Docusauros](https://docusaurus.io) 같은 `Static Site Generator`들을 사용하고 있고, 대개의 상황에 해당하는 다중 Project Repository를 관리하여 사용할 경우에는 [Antora](https://antora.org)를 이용한다. *(Antora의 경우 [Asciidoctor](https://asciidoctor.org/)을 사용해야 한다)*

여기에 설계문서를 작성할 떄 다이어그램을 *(e.g., [Graphviz](https://graphviz.org), [PlantUML](https://plantuml.com))* 위해 [Kroki](https://kroki.io)를 함께 사용하는 경우가 일반적이다. *(추가적으로 DNS나 TLS/SSL을 위한 Web Server까지)*

요즘은 많은 회사들이 개발자들을 전문적으로 지원하기 위해 [Swagger UI](https://swagger.io/tools/swagger-ui)처럼 적용 및 실행 등 더 많은 기능을 `Platform Engineer`를 통해 `Developer Portal`를 구축하는 것 같다. *(e.g., [Backstage](https://backstage.io))*

이렇게 도큐먼트들을 관리하는 Application은 원하는 테마, internal/external 여부, 확장된 기능, 타 개발환경 내의 integration 등 용도에 맞게 선택하여 사용해왔다.

특히 DaC 문서들을 노출하는 때에는 몇가지 이유가 있었다. *(i.e., 외부 호스팅하는 이유)*
1) SEO나 분석을 위해. *(e.g., [Google Search Console](https://search.google.com/search-console), [Google Analytics](https://developers.google.com/analytics))*
2) 문서 작성을 쉽게 improvise하기 위해. *(e.g., [Gitlab Web IDE](https://docs.gitlab.com/user/project/web_ide))*
3) 외부에 노출되는 창피함을 품질 강화의 기회로 활용하기 위해. *(지극히 개인적인 이유다)*
4) 더 많고 다양한 피드백을 받기 위해.
5) 자신의 PR을 위해. *(프로젝트 포트폴리오보다는 지속적인 활동을 부각하는데 더 낫다고 생각하며, 보통 지원서에 URL을 요청하는 곳이 믾았다)*

그래서 [성장일지 포스팅](https://coldnine.github.io)에서는 [GitLab Pages](https://docs.gitlab.com/user/project/pages)를, [기술 블로그](https://cold9.gitlab.io/blog)에서는 [Gitlab Pages](https://docs.gitlab.com/user/project/pages)를 이용하여 간단한 수준의 CI를 덧붙여 호스팅을 관리하고 있다. *(Gitlab Pages와 달리, Github Pages에서는 private repository를 호스팅 할 수 없었던 것으로 기억한다)*

이런 서비스들은 [Jekyll](https://jekyllrb.com), [Hugo](https://gohugo.io), [Gatsby](https://www.gatsbyjs.com/) 등의 Ruby 기반 `Static Site Generator`을 이용하면 쉽게 작성이 가능하다. *(c.f., [Reddit 글](https://www.reddit.com/r/webdev/comments/b0j9rs/infographic_gatsby_vs_hugo_vs_jekyll/?rdt=63820))*

그 중 성장일지를 위한 Jekyll 사용방법은 [README](https://github.com/ColdNine/coldnine.github.io)에 작성하였다.

마지막으로 [Google Trends](https://trends.google.com/trends/explore?date=all&q=Document%20as%20Code)를 인용하여 `DaC`를 권장하는 것으로 성장일지를 마친다.
